# TRIZEL Canonical Metadata Templates

**Version:** 1.1.0  
**Status:** Canonical, Archival-Ready  
**Last Updated:** 2026-01-12

## Overview

This directory contains the canonical metadata templates used across TRIZEL publications and research outputs. These templates enforce strict governance compliance, validation requirements, and multi-language support for sustainable archival and publication workflows.

## Files

### 1. CANONICAL_METADATA_TEMPLATE.yaml
**Status:** Single Source of Truth

This YAML file is the **authoritative template** for all author and publication metadata across TRIZEL repositories. It defines the standard schema and default values for:
- Author information (name, ORCID, affiliation)
- Publication metadata (title, description, creators, contributors)
- Licensing and access rights
- Keywords and related identifiers
- Zenodo/DataCite compatible fields
- Multi-language support (EN, FR, AR fully; ZH planned)
- Governance and validation requirements

### 2. CANONICAL_METADATA_TEMPLATE.json
**Status:** Mirror (Derived)

This JSON file is a **faithful mirror** of the YAML template, provided for consumers and systems that require JSON format. It includes a JSON Schema reference for automated validation.

**Important:** Do NOT edit this file directly. It is automatically regenerated from the YAML source when updates are needed.

## Governance Policies

### Authority and Control
1. **Single Source of Truth**: The YAML template is the canonical source
2. **Version Control**: All changes must be tracked via Git with semantic versioning
3. **Immutability**: Historical versions are preserved and never deleted
4. **Approval Process**: Metadata schema changes require explicit approval
5. **Non-Interpretive**: Metadata is factual and descriptive only

### Data Classification and Protection
- **Classification Levels**: open | internal | confidential | restricted
- **Default Classification**: open (for public research)
- **Retention Policy**: permanent (archival-ready)
- **Version Control**: required for all metadata instances
- **Access Control**: Governed by repository-level permissions

### Compliance Requirements
- **ORCID**: Required for all creators and contributors
- **Licensing**: Explicit license specification required before publication
- **DOI**: Recommended for all archival publications
- **Data Protection**: No personal data beyond professional affiliations

## Usage Policy

### Editing Guidelines
1. **YAML is the single source of truth** — All metadata changes must be made to the YAML file first
2. **JSON is a derived mirror** — After editing YAML, regenerate the JSON file to maintain parity
3. **Never edit JSON directly** — Manual JSON edits will be overwritten during regeneration
4. **Maintain field parity** — Both files must contain identical field structure and values
5. **Follow semantic versioning** — Update schema_version for breaking changes

### Regeneration Process
When updating metadata:
1. Edit `CANONICAL_METADATA_TEMPLATE.yaml`
2. Validate YAML syntax
3. Regenerate `CANONICAL_METADATA_TEMPLATE.json` from YAML
4. Verify JSON structure matches YAML content
5. Update version numbers in both files
6. Commit both files together with descriptive message

### Reusability
These templates are designed to be reused across:
- Multiple TRIZEL repositories
- Publication systems (Zenodo, DataCite, etc.)
- Archive and preservation workflows
- DOI registration processes
- Multi-language publication platforms

## Validation Rules

### Required Fields
- `author.name`: Must be non-empty string
- `author.orcid`: Must match ORCID format (XXXX-XXXX-XXXX-XXXX)
- `publication_metadata.title`: Must be present before publication
- `publication_metadata.version`: Must follow semantic versioning (X.Y.Z)
- `publication_metadata.creators`: Must include at least one creator with ORCID
- `publication_metadata.license`: Must be specified before publication

### Optional but Recommended
- `publication_metadata.keywords`: At least 3-5 keywords recommended
- `publication_metadata.related_identifiers`: Link to related works (DOI, URL)
- `publication_metadata.description`: Comprehensive abstract (150-300 words)

### Field Format Validation
- **Dates**: ISO 8601 format (YYYY-MM-DD)
- **ORCIDs**: Format XXXX-XXXX-XXXX-XXXX with optional https://orcid.org/ prefix
- **DOIs**: Format 10.XXXX/XXXXX with optional https://doi.org/ prefix
- **URLs**: Valid HTTPS URLs only
- **Languages**: ISO 639-1 two-letter codes

### Automated Validation
Metadata instances should be validated against the schema before:
- Submission to publication platforms
- Archival deposit
- DOI registration
- Cross-repository replication

## Deprecated Terms Policy

### Prohibition of Deprecated Fields
The following practices are **explicitly disallowed**:

1. **Legacy Field Names**:
   - `author_name` → Use `author.name`
   - `pub_date` → Use `publication_date`
   - `langs` → Use `languages_supported`

2. **Ambiguous Terms**:
   - Avoid generic "owner" → Use specific role (creator, contributor)
   - Avoid "private" → Use `access_right: closed` or `restricted`

3. **Non-Standard Formats**:
   - No custom date formats → ISO 8601 only
   - No abbreviated ORCIDs → Full 16-digit format required
   - No bare DOIs → Include full https://doi.org/ prefix

4. **Maintaining Deprecated Terms List**:
   - Track deprecated terms in `template_info.deprecated_terms[]`
   - Update list when schema evolves
   - Never reuse deprecated field names

## Multi-Language Support

### Supported Languages

#### Fully Supported (Production)
- **English (EN)**: Primary language, Latin script
- **French (FR)**: Full support, Latin script
- **Arabic (AR)**: Full support, Arabic script (RTL)

#### Planned Support (Future)
- **Chinese (ZH)**: Planned, Han script (Simplified/Traditional)

### Language Metadata Structure
```yaml
languages_supported:
  - code: "en"
    status: "primary"
    script: "Latin"
  - code: "fr"
    status: "supported"
    script: "Latin"
  - code: "ar"
    status: "supported"
    script: "Arabic"
  - code: "zh"
    status: "planned"
    script: "Han"
```

### Translation Fields
- `title_translations`: Translated titles by language code
- `description_translations`: Translated descriptions by language code
- `keywords`: Can include multilingual terms

### Script Support
- **Latin**: EN, FR
- **Arabic**: AR (right-to-left rendering required)
- **Han**: ZH (simplified/traditional variants)

### Character Encoding
- All files use UTF-8 encoding
- RTL markers preserved for Arabic content
- Bidirectional text (Bidi) support required for mixed scripts
