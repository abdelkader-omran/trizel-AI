PUBLISHING RULES AND PROVENANCE STANDARD
=======================================

Project : TRIZEL Epistemic Operational Portal
Target  : https://trizel-ai.com/
Phase   : FINAL_EXECUTION (Phase-5)
Spec ID : PUBLISH-PROVENANCE-v1.0
Status  : Governance-locked

----------------------------------------------------------------
1. PURPOSE
----------------------------------------------------------------
This document defines the mandatory rules governing what may be
published on the TRIZEL portal and how provenance must be preserved.

These rules are normative and deploy-blocking.

----------------------------------------------------------------
2. FUNDAMENTAL PRINCIPLE
----------------------------------------------------------------
The TRIZEL portal does not publish content.
It exposes declared, versioned, and traceable outputs.

Any content lacking provenance is treated as non-existent.

----------------------------------------------------------------
3. ELIGIBLE CONTENT (DECLARED ONLY)
----------------------------------------------------------------
The portal MAY expose only artifacts that satisfy ALL of the following:

- explicitly declared in outputs.index.json
- version-locked (commit hash or release tag)
- generated by an identified repository
- assigned an epistemic status
- reproducible from the source repository

No implicit inclusion is permitted.

----------------------------------------------------------------
4. PROHIBITED CONTENT (ABSOLUTE)
----------------------------------------------------------------
The following content MUST NEVER be exposed:

- undeclared artifacts
- intermediate calculations
- draft or working documents
- speculative or interpretative text
- internal notes or correspondence
- AI-generated narrative or summaries
- screenshots used as evidence
- placeholder or demo material

Any violation blocks deployment.

----------------------------------------------------------------
5. PROVENANCE REQUIREMENTS
----------------------------------------------------------------
Every exposed artifact MUST include the following metadata:

- repository identifier
- repository role (producer | integrator)
- artifact name and type
- version or commit reference
- generation timestamp
- epistemic status (Stable | Beta | In Progress)

Absence of any field invalidates eligibility.

----------------------------------------------------------------
6. ORPHAN CONTENT PROHIBITION
----------------------------------------------------------------
Content without verifiable provenance is forbidden.

Manual insertion of content into the portal repository is prohibited
unless it is governance documentation.

----------------------------------------------------------------
7. MODIFICATION AND DERIVATION RULES
----------------------------------------------------------------
The portal MUST NOT modify exposed artifacts.

- no reformatting that changes meaning
- no summarization
- no extraction of partial results
- no recombination outside integrator outputs

All interpretation remains upstream.

----------------------------------------------------------------
8. VERSION LOCKING
----------------------------------------------------------------
Once an artifact is exposed:

- its version MUST remain fixed
- updates require a new version
- prior versions remain accessible for audit

Retroactive modification is forbidden.

----------------------------------------------------------------
9. EPISTEMIC STATUS DISCIPLINE
----------------------------------------------------------------
Each artifact MUST be labeled with exactly one status:

- Stable        : validated and frozen
- Beta          : exposed with known limitations
- In Progress   : incomplete but traceable

Status inflation or ambiguity is prohibited.

----------------------------------------------------------------
10. AUDIT VISIBILITY
----------------------------------------------------------------
The following MUST be publicly visible:

- list of exposed artifacts
- their provenance metadata
- build timestamp
- portal commit hash
- validation outcomes

Hidden publication logic is forbidden.

----------------------------------------------------------------
11. ENFORCEMENT
----------------------------------------------------------------
Violation of any rule in this document results in:

- immediate deployment blockage
- recording of the violation in gate-report metadata
- required corrective commit before retry

----------------------------------------------------------------
12. VERSIONING
----------------------------------------------------------------
This standard is version-locked.

Any modification requires:
- governance approval
- version increment
- explicit commit reference

----------------------------------------------------------------
AUTHORITY
----------------------------------------------------------------
This document is binding for Phase-5 FINAL_EXECUTION.
Non-compliance invalidates deployment eligibility.
